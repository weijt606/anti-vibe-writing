# 场景预设

不同的写作场景有不同的"硬约束"：长度、结构、是否允许 emoji、读者注意力的尺度。这份预设把六种常见场景的约束写清楚。

预设和 host profile 配合使用：场景预设决定形态，host profile 决定声音。两者不冲突时叠加，冲突时按 [learning-mode.md](./learning-mode.md) 末尾的优先级处理。

## 1. 推特 / X.com 发帖

**形态约束：**
- 单条 280 字符内（英文）/ 140 字内（中文体感节奏）
- 串帖（thread）每条独立成立，第一条要能单独读
- 不分标题层级
- 不用 markdown 列表（平台不渲染）
- 链接和 @mention 节制使用

**写作约束：**
- 第一句必须钩住读者：观点、反常识、问题、画面感
- 不要"今天分享一下"、"接下来讲讲"、"先说结论"这种引导语
- 每条留白：不要把所有信息塞进一条，留下一条想继续读的理由
- 可以用语气词、可以倒装、可以不完整句
- 数据放具体数字，不要"很多 / 一些 / 大量"

**禁忌：**
- 长段落
- "干货分享"、"硬核"、"建议收藏"等流量套话
- 一条结尾说"详见下一条"，下一条自己开始就好

**结尾的几种方式：**
- 抛一个反问
- 给一个具体下一步
- 留一句让人想转发的话
- 干脆停在事实

**（可选）随手感：** 用户明确要"像手机随手发的"时，可启用人味儿质感模式里的"随手打"子档：漏句末标点、不大写、漏个虚词，但只在休闲口吻，且绝不碰数字 / 名称 / 链接，见 [human-texture.md](./human-texture.md)。默认不开。

## 2. 微博 / 即刻 / 小红书

**形态约束：**
- 中文社交平台节奏比推特慢一点，可以多两三段
- 即刻接受较长内容（500–1500 字），微博偏短
- 小红书需要笔记封面意识，首段要"标题感"
- emoji 和颜文字在小红书是常态，在即刻偏克制，在微博看圈层
- hashtag / 话题标签按平台习惯

**写作约束：**
- 首句要么是场景（"刚刚在地铁上想到"），要么是判断（"做了一年才发现这事错了"）
- 多用换行让段落破碎，长段落在手机屏上没人读
- 允许"小标题感"的短句单独成段
- 口语化优于书面化

**禁忌：**
- 公众号那种"导语 + 一二三四 + 总结"的结构
- 自称"小编"、"本博主"
- 三连排比

**学习模式重点：**
- 看用户在哪个平台、什么圈层，风格差异极大
- 提取用户的 emoji 习惯（用 / 不用 / 用哪几个）

**（可选）随手感：** 同推特，休闲短贴可启用"随手打"子档（默认关）。小红书笔记若偏精致排版则不建议用。

## 3. 博客 / 公众号 / Newsletter

**形态约束：**
- 长度 1500–4000 字常见
- 允许 H2 / H3 小标题
- 可以放代码块、图、引用
- 公众号注意分段密度（手机阅读，段不宜过长）

**写作约束：**
- 开头三段决定读者要不要往下看
- 论点要早出现，论据后展开，不要"先讲背景"
- 每个小标题要承担信息，不要只做"装饰"
- 例子要具体到能复现：哪个产品 / 哪个版本 / 哪个时间 / 哪个数字
- 收尾要落到"读者能拿走什么"，不要升华到行业未来

**禁忌：**
- "话不多说，直接开始"：直接开始就好
- "希望本文对你有所帮助"
- 文末"如果你觉得有用，欢迎点赞 / 关注 / 转发"塞太多
- 用 AI 起的开头："在 X 快速发展的今天……"

**学习模式重点：**
- 用户怎么开头（场景 / 观点 / 问题 / 反常识）
- 用户用什么类型的例子（个人经历 / 公开案例 / 数据）
- 用户怎么收尾（具体下一步 / 留思考 / 半句话停住）

## 4. 播客 show notes / 视频脚本

**形态约束：**
- show notes 通常 300–800 字 + 时间戳
- 视频脚本看长度，口播稿要标停顿
- 允许口语化标记（"嗯"、"对"、"那"）但节制使用

**写作约束：**
- 模拟说话节奏：句子比博客短，重复可以多一点
- 时间戳要点出真正值得跳到的那一刻，不要每分钟标一个
- 描述嘉宾时要避免"业内资深"、"重磅嘉宾"这类套话
- 引用嘉宾观点时尽量原文，不要"嘉宾认为"地复述

**禁忌：**
- 把口播稿写得像论文
- 给嘉宾贴一堆头衔
- "本期节目精彩纷呈"、"干货满满"

**学习模式重点：**
- 用户是否爱用"嗯 / 对 / 那"等口语标记
- 用户怎么处理停顿（破折号 / 省略号 / 句号 + 换行）
- 用户的 show notes 是流水账型还是论点型

## 5. 专业报告 / 技术备忘 / 内部文档

**形态约束：**
- 结构要严密，可以分层小标题
- 允许长句、长段
- 数据和引用要可核查（链接 / 文献 / 内部 ticket 编号）
- 不允许 emoji
- 允许专业术语，但首次出现要简短说明（除非读者明确是同行）

**写作约束：**
- 结论要明确，不接受"具体问题具体分析"式收尾
- 不确定性要量化，不要藏在"或许 / 可能 / 一定程度上"里
- 多用"如果 X，则 Y；否则 Z"的明确分支
- 取舍要写出"我们选了什么 / 放弃了什么 / 为什么"

**禁忌：**
- 中性化到不下判断
- "建议综合考虑各方面因素"
- 三连排比
- 把数据塞进表格但不解读

**学习模式重点：**
- 用户怎么表达不确定（保守 / 直接 / 量化）
- 用户多长的句子能维持准确而不啰嗦
- 用户在专业语境里的"个人标志"：是否有特定的论证套路

## 6. Reddit / 英文论坛评论

**形态约束：**
- 评论是"帮人"，不是发广告。读者一眼能闻出推销味，闻到就划走或举报
- 不分标题层级，不用花哨 markdown；最多简单换行、一两个要点
- 一条评论一个链接封顶，多数情况下一个都不放
- 多为英文，语气随子版块（sub）调：技术创业向和老师 / 教育向的口吻完全不同

**写作约束：**
- 给具体的东西：三条能照做的标准或做法，比空泛的"要多互动"强得多。带一个真实、具体到有点怪的细节（真发生过的，不许编）
- 用口语连接词：so、honestly、anyway、tbh，句长要参差
- 先亮一个真实观点，凭观点挣 upvote，别和稀泥
- 不写引导语（"here are three things..."），不堆咨询词（leverage、streamline、robust）
- 提到竞品别踩，"I've used X and Y in different contexts" 这种中性说法没问题

**禁忌（很多会被自动拦或被踩）：**
- **破折号（`—` / `——`）一个都别用。** 一些子版块的 automod 现在把破折号密度当成 AI 信号，据此把评论当 "low-effort / AI" 自动删掉。改用逗号、句号，或 so / and 接住
- 太工整的 "it's not X, it's Y" 对仗、左右完全对称的段落。把它打散，长短句混着来
- 装成普通用户替自己产品背书："I had the same problem! I've been using [自家产品]..."。这是 sock-puppet 形状，基本会被识破
- 一天内在多个 sub 重复同一套自我介绍，或原样粘贴同一段模板
- 不披露利益关系的软广

**诚实与边界（底线）：**
- 提到自己有关系的产品，主动披露
- 模板每次都改成自己的话；原样粘贴会被认出来
- 读着不像自己会说的话，就别发

这一条不是"教你躲过 automod 发广告"。是反过来：写一条真有价值、真像人说的评论，顺带不被规则误伤。靠造假冲量迟早被社区反噬。

## 跨场景的不变量

不管哪个场景，下面几条不变：

- 不编事实
- 不替原作者强化立场
- 保留必要的限定词
- 数字、名称、引用不能改
- 不用"赋能 / 打通 / 闭环 / 抓手"

场景变，约束变，但反 AI 味的底线不变。
